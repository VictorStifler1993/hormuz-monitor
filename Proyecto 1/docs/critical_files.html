Archivos Críticos del Sistema

Qué son, qué función cumplen, y por qué son críticos

¿Qué es un archivo crítico? Es un archivo que, si se elimina, se corrompe o tiene un bug, rompe múltiples partes del sistema en cascada. No todos los archivos son igual de importantes: un scraper puede fallar y los demás siguen funcionando. Pero si falla un archivo crítico, todo se para. Son los "puntos de fallo único" del sistema.
Piensa en una fábrica: si se rompe una máquina de empaquetado, puedes empaquetar a mano. Pero si se va la luz (el generador central), toda la fábrica se detiene. Los archivos críticos son el generador de esta fábrica.

shared/models.py COMPARTIDO
Qué hace Define las estructuras de datos fundamentales del sistema: RawArticle, ClassifiedArticle, OilPrice, CorrelationResult. Son las "plantillas" que describen cómo se ve un artículo, un precio, una clasificación. Todos los módulos del sistema hablan este mismo idioma.
Por qué es crítico Es la lengua común de todo el sistema. Cada módulo — scrapers, clasificador, base de datos, analizadores, notificadores — importa estos modelos. Si cambias un campo aquí, afecta a 15+ archivos simultáneamente. Es el archivo con más "dependientes" de todo el proyecto.
Qué se rompe si falla TODO. Los scrapers no pueden crear artículos. La base de datos no puede guardarlos. El clasificador no puede devolver resultados. Los analizadores no pueden procesar datos. Los informes no se generan. El sistema entero queda inoperativo.
cloud/main.py CLOUD
Qué hace Es el director de orquesta del componente cloud. Cada 15 minutos: activa los scrapers, filtra duplicados, guarda los artículos nuevos en JSONL, y comprueba si hay alertas por keywords. Es el archivo que GitHub Actions ejecuta directamente.
Por qué es crítico Sin este archivo, el sistema deja de recopilar datos automáticamente. Es el punto de entrada que conecta el cron de GitHub Actions con toda la lógica de scraping. Si falla, no llegan datos nuevos, y sin datos nuevos, todo el análisis local se queda estancado con información vieja.
Qué se rompe si falla La recopilación automática de noticias se detiene. No se generan archivos JSONL nuevos. Las alertas por keywords dejan de funcionar. El componente local no tiene datos frescos que analizar. Es como cortar el suministro de agua a la planta de tratamiento.
local/classifier/subagent_classifier.py LOCAL
Qué hace Toma artículos en bruto y los envía a un subagente de Claude Code para clasificarlos: ¿es escalada o desescalada? ¿Cuán relevante es? ¿Qué actores están involucrados? ¿Cómo podría afectar al precio? Transforma texto plano en datos estructurados y cuantificados.
Por qué es crítico Es el cerebro del sistema. Sin clasificación, los artículos son solo texto sin procesar. No puedes correlacionar "texto" con precios del petróleo. Necesitas los números (escalation_score, relevance_score) para que toda la cadena de análisis funcione. Es lo que convierte noticias en señales cuantificables.
Qué se rompe si falla Todos los módulos de análisis quedan sin datos. No hay correlaciones. No hay backtesting. No hay descubrimiento de patrones. No hay notificaciones por email (porque las notificaciones dependen del score de relevancia y escalación). El sistema recopila noticias pero no las entiende.
.github/workflows/hormuz_monitor.yml INFRAESTRUCTURA
Qué hace Es la configuración de GitHub Actions que programa la ejecución automática cada 15 minutos. Define: qué versión de Python usar, qué dependencias instalar, qué secretos inyectar (API keys, credenciales de email), y cómo commitear los datos nuevos de vuelta al repositorio.
Por qué es crítico Es lo que hace el sistema autónomo. Sin este archivo, tendrías que ejecutar el scraper manualmente cada 15 minutos, 96 veces al día, 24/7. Este archivo es la diferencia entre un script que corres a mano y un sistema que trabaja solo mientras duermes.
Qué se rompe si falla El monitoreo automático se detiene completamente. No se recogen noticias mientras no estés delante del ordenador. Puedes perder eventos críticos que ocurran de madrugada o en fin de semana. El sistema pasa de ser un monitor 24/7 a ser un script manual.
local/analysis/pattern_discovery.py LOCAL
Qué hace Es el motor de "lo que no sabemos que no sabemos". Busca: movimientos de precio sin noticias que los expliquen, términos nuevos emergentes, cambios de régimen en la relación noticias-precio, causalidad de Granger, y efectos de interacción entre categorías.
Por qué es crítico Es lo que diferencia este sistema de un simple agregador de noticias. Cualquiera puede hacer un scraper de noticias. Lo que hace valioso a este proyecto es su capacidad de descubrir patrones ocultos y relaciones no obvias. Sin este módulo, el sistema solo recopila y clasifica — no descubre.
Qué se rompe si falla Se pierde la capacidad de descubrir anomalías, cambios de régimen y causalidad. El sistema sigue funcionando (scraping, clasificación, correlación básica), pero pierde su ventaja competitiva: la exploración. Es como tener un microscopio y solo usarlo como lupa.
local/analysis/scale_calibrator.py LOCAL
Qué hace Analiza qué scores de escalación realmente precedieron movimientos de precio y ajusta los umbrales automáticamente. Con el tiempo, la escala se adapta a la realidad en lugar de usar categorías fijas predefinidas.
Por qué es crítico Es lo que hace al sistema adaptativo. Los mercados cambian. Lo que movía el precio del petróleo hace 6 meses puede no moverlo hoy. Sin calibración, el sistema usa umbrales que pueden volverse obsoletos, generando falsos positivos (alertas innecesarias) o falsos negativos (eventos ignorados que sí importaban).
Qué se rompe si falla El sistema sigue funcionando pero con una escala potencialmente desactualizada. Las notificaciones pueden llegar por eventos irrelevantes o no llegar por eventos importantes. La calidad del análisis se degrada gradualmente con el tiempo.
